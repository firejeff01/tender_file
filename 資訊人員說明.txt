■ 設計理念（為什麼這樣做）
────────────────────────────────────────────────────────────────────
需求限制：公司網管環境不允許架站台、不允許安裝檔，行政電腦只有
Windows 11 內建元件。因此本工具刻意做到「零安裝、零外部相依」：

  ・目標框架 .NET Framework 4.8。這是 Windows 10 1903+ 與 Windows 11
    的「作業系統內建元件」，不是需要另外安裝的 runtime。
  ・以 Windows 內建的 C# 編譯器 csc.exe 建置，不需要 Visual Studio、
    不需要 dotnet SDK、不需要 NuGet、不需要連外網。
  ・執行期完全不依賴 Microsoft Office 或 LibreOffice：xlsx 與 odt
    都用內建的 System.IO.Compression（zip）＋ System.Xml 直接處理。
  ・整包就是一個資料夾，複製到哪都能跑，也可放共用磁碟。

■ 如何重新建置（例如防毒誤刪 exe、或修改原始碼後）
────────────────────────────────────────────────────────────────────
  1. 確認「原始碼」資料夾內有 *.cs 與 app.manifest。
  2. 雙擊「建置.bat」。它會呼叫：
       %WINDIR%\Microsoft.NET\Framework64\v4.0.30319\csc.exe
     編譯出「標案文件產生器.exe」。
  3. 若公司用 AppLocker／WDAC 白名單擋掉未簽章 exe：
     由 IT 用建置.bat 在受信任的路徑重新編譯，或把該路徑加入白名單。
     原始碼隨附的用意就是讓 IT 能自行稽核與重建，不必信任外來 binary。

  建置需求：任何 Windows 10/11（csc.exe 是 .NET Framework 的一部分，
  預設就在系統裡）。若 Framework64 路徑不存在，建置.bat 會自動改用
  32 位元路徑 Framework\v4.0.30319\csc.exe。

■ 原始碼結構
────────────────────────────────────────────────────────────────────
  Program.cs      進入點。一般開 GUI；引數 /selftest 跑自我測試。
                  含全域未處理例外攔截（顯示可讀對話框而非崩潰視窗）。
  MainForm.cs     WinForms 主視窗（清單、勾選、產生、結果回報、輸出資料夾設定、新增範本）。
  Settings.cs     可攜式設定讀寫（exe 旁「設定.txt」，key=value，UTF-8）。
                  目前只存「輸出資料夾」；刪掉設定.txt 即還原預設（exe 旁的「輸出」）。
  TemplateBuilder.cs 「新增範本」核心：解析任意 ODT、擷取彩色 span（非黑色 fo:color）、
                  相鄰同色 span 合併、套用參數對應→掛 PARAM.* 樣式→重用 OdtWriter.Repack。
  XlsxEditor.cs   純 zip+XML 對既有 標案資料.xlsx 加欄（標案清單 ListObject 表格）／加列
                  （公司資料）。備份→暫存→重讀驗證→取代，失敗還原，不依賴 Office。
  AddTemplateService.cs 「新增範本」無 UI 流程（Commit）：正規化→更新 Excel→存範本→重寫 tokens.txt。
  AddTemplateForm.cs 「新增範本」對話框（對應表、智慧預填、新增參數）。

  「新增範本」與原本 工具\normalize.ps1 的關係：normalize.ps1 是最初一次性把 3 份範本
  正規化（示範文字寫死）；TemplateBuilder 是它的通用化 in-app 版本，改以「使用者對應
  彩色片段」辨識參數，替換/打包規則完全相同。日後新範本一律走「新增範本」按鈕即可，
  不需再跑 normalize.ps1。
  XlsxReader.cs   純 zip+XML 的 xlsx 讀取器（不依賴 Office）。
  OdtTemplate.cs  範本載入、參數替換、確定性 zip 重打包（見下）。
  Generator.cs    Excel→產生計畫、資料夾名清理、逐列產生。
  Validator.cs    產出 ODT 的結構自我驗證。
  Util.cs         字串正規化、資料夾名安全化、民國日期解析。
  SelfTest.cs     /selftest 的測試內容（45 項）。
  app.manifest    System DPI aware、asInvoker、supportedOS。

  工具\normalize.ps1   一次性：把原始範本正規化成 ${參數} 版（見下）。
  工具\make_xlsx.ps1   一次性：用 Excel COM 產生初始 標案資料.xlsx。
  （這兩支只在準備範本/Excel 時於開發機執行，不隨部署包給行政人員。）

■ 關於 ODT 重打包（重要技術點）
────────────────────────────────────────────────────────────────────
  ODF 規範要求 mimetype 必須是壓縮檔的第一個項目、且以 stored（未壓縮、
  無 extra field）儲存。國發會的 ODF 文件格式檢測工具會驗這一點。

  .NET Framework 的 ZipArchive 即使指定 NoCompression，仍會以 Deflate
  method 寫入，不符規範。因此 OdtTemplate.cs 的 Repack() 自行輸出
  local header / central directory / EOCD（含 CRC32、DOS 時間），
  確保 mimetype 為 stored 且第一。每次產生後 Validator 會重新讀回驗證。

■ 範本正規化（若日後要新增或修改範本）
────────────────────────────────────────────────────────────────────
  範本內用 ${參數名} 標記可替換欄位，例如 ${廠商名稱}、${機關名稱}、
  ${簽署年}${簽署月}${簽署日}。每個參數 span 掛專屬樣式 PARAM.xxx。
  範本資料夾內的 tokens.txt 記錄每個範本各參數的預期出現次數；程式
  啟動時會據此預檢，若被 Word/LibreOffice 重存導致 span 被拆散、
  次數對不上，會拒絕產生並提示「範本疑似被編輯過」。

  要新增範本：把原始 ODT 放進來，在其中把要參數化的文字改成 ${參數名}
  並套上紅色（或直接參考現有範本做法），再重跑 工具\normalize.ps1
  重新產生凍結版與 tokens.txt。normalize.ps1 目前針對這 3 個特定範本
  的示範資料（哈瑪星、陳文德等）做雙軌掃描，換新範本時需調整腳本內的
  字面值清單。

  對應 Excel 欄位：程式會掃「範本\*.odt」的所有 ${X}，X 就是 Excel
  需要的欄名或「公司資料」的參數名。「產生:<範本檔名(不含.odt)>」欄
  控制每列要產哪些文件。

■ 自我測試
────────────────────────────────────────────────────────────────────
  在本資料夾開命令列執行：
       標案文件產生器.exe /selftest
  會產生 selftest.log 並以 exit code 回報失敗數（0 = 全過）。
  涵蓋：日期/資料夾名/科學記號等單元測試、xlsx 解析邊角、端到端產生
  （XML 跳脫、換行、超長/非法/保留資料夾名、重複列、略過/覆寫、
  紅黑顏色、缺參數、範本竄改偵測）。

■ 系統需求
────────────────────────────────────────────────────────────────────
  ・執行：Windows 10 1903 以上 或 Windows 11（皆內建 .NET FW 4.8）。
  ・建置：同上，使用系統內建 csc.exe。
  ・準備範本/Excel（normalize.ps1、make_xlsx.ps1）：需要 PowerShell；
    make_xlsx.ps1 需要安裝 Excel（僅開發機一次性使用）。
